# Code Review

**Code Review** - процесс проверки кода другим разработчиком

- Основная цель - не поиск багов (это работа тестировщиков), а **читаемость, понятность, качество решений**
- Хорошее ревью = подсказки, что улучшить, какой подход лучше, что стоит почитать
- Меньше 15-20 минут - поверхностная проверка. Норма от 30 минут, сложные задачи - значительно дольше
- Обратная связь будет оценена тем, кто её получает - плохая история может стоить рекомендаций или места в команде

# Sandwich Feedback

Замечания «упаковываются» в три слоя:

**1. Верхняя булка - Похвала** Начинаем с чего-то хорошего. Отмечаем что получилось, что сделано правильно или изящно.

**2. Начинка - Критика / Правки** Суть ревью - конкретный список того, что нужно исправить. Это сухая карточка/чек-лист без перехода на личности, без «ну это же так очевидно» и т.п. Её можно передать другому коллеге - и он поймёт что делать, даже если изначально писалась не для него.

**3. Нижняя булка - Поддержка / Итог** Заканчиваем на позитивной ноте или общем векторе развития. Чтобы не осталось ощущения «я плохой кодер».

> **Минус метода:** если «булки» каждый раз одинаковые и приторные - разработчик начинает их игнорировать и сразу ищет глазами критику

# Структура ревью

## 1. Плюсы

- Строго по теме - не нужно здесь писать про баги или рекомендации
- Хвалим, подбадриваем, отмечаем что получилось - помогаем поддержать мотивацию
- Разговариваем как нормальные люди, а не короткими фразами и сухими списками
- Если сами чему-то научились у автора - делимся этим

**Плохие примеры:**

- `к сожалению не нашел` - человека нужно подбодрить, а не отпустить ему руки
- `все норм` - нейтрально, не подбадривает
- `ну есть две папки` - похвала с надменностью, на грани токсичности
- `все круто` - лучше чем ничего, но можно конкретнее

**Хорошие примеры:**

```
+ Кнопки пагинации имеют hover-эффект
+ Используются css-переменные
+ Адаптив до 1024px не ломается
+ Реализован компонент Icon
+ Утилитарная логика отделена от логики компонентов
```

```
Хорошая работа. Молодец!
- использование семантических тегов
- использованы display: flex & grid в css
- логически верная разбивка на компоненты
- интересная реализация задачи поиска
- использование css-переменных
```

```

По традиции вот список плюсов:
1. Используются и выделены в отдельное место презентационные компоненты
```

## 2. Баги

- Никаких оценок, суждений, эмоций, переходов на личность и токсичной коммуникации
- Проверка строго по ТЗ
- Чёткий и конкретный список — по факту это ТЗ на дебагинг
- Прикладываем скриншоты, примеры кода, ссылки — это сильно ускоряет фикс
- Из описания бага должно быть понятно как его воспроизвести
- Разделяем **критические** (нарушение ТЗ) и **некритические** (мелкие визуальные) баги
- 3-6 некритических багов ≈ 1 критический — сильно занижать оценку из-за мелочей не стоит

**Плохие примеры:**

- `-` (прочерк при невысокой оценке) — 0 информации
- `крестик не работает` — нет контекста, нет пруфов, непонятно где и что
- `а где адаптив планшета?? Нет от слова совсем` — надменность, оценочность
- `Огромное количество багов. Описывать все нет смысла` — 0 полезности
- `плохая файловая структура` — это не баг если ТЗ соблюдено, выносить в рекомендации
- `тупо вызывают методы` — оценочность в формулировке, так не стоит

**Хорошие примеры:**

```
Критические баги:
- forecast-weather не имеет отступ вниз на 8px (фиксится через margin-bottom: 8px)
- Сломан путь у <img src="./img/icons/weather-type/day-cloudy.svg">,
  должен быть src="./img/weather-type/day-cloudy.svg"

Некритические баги:
- В поле поиска не меняется иконка на крестик когда поле в фокусе
- После отправки текста по Enter, текст остаётся в поле поиска
- Цвет стрелки у правого переключателя отличается от макета (рис. 1)
```

```
Смотри, в ТЗ есть такая строчка:
> После выбора количества вопросов и нажатия «Начать» — сформировать квиз
> с заданным числом рандомных вопросов

У тебя сейчас вопросы выбираются по порядку из json-файла.
Нужно предусмотреть утилиту, которая перемешивает и возвращает массив
с заданным количеством вопросов.
```

## 3. Рекомендации

- Даём развёрнутый ответ, а не сухую фразу-отписку
- Общаемся так, как сами хотели бы чтобы общались с нами более опытные коллеги
- Аргументируем и приводим примеры — мнение против мнения это дискуссия, не рекомендация
- Делимся ссылками и полезными материалами — очень приветствуется
- Рекомендация = делимся опытом и советами, не навязываем точку зрения

Хорошие формулировки:

- `как мне кажется...`
- `можно было бы...`
- `я использую такой подход там-то...`
- `возможно имеет смысл посмотреть вот это...`
- `вот ссылка на статью по теме...`

**Плохие примеры:**

- `нечего` — 0 информации
- `все хорошо` — ничем не помогает
- `устранить баги` — это не рекомендация
- `думаю стоит купить очки, ибо верстка с макетом не совпадает вообще` — недостойная коммуникация
- `все svg лучше наверное в отдельные файлики` — без обоснования почему

**Хорошие примеры:**

```
Весь JS лучше сложить в отдельную папку — будет аккуратнее и удобнее в поддержке.
Советую почитать про БЭМ и следовать такому неймингу — это очень хорошая практика.
Такие вещи как °, % и подобное лучше верстать через ::after.
```

```
Признаюсь, даже не знаю что тут посоветовать.
Видимо, сам скоро начну обращаться за советом к тебе. Так держать!
```

_(отличная рекомендация если попалась работа сильнее твоей)_

# Общие правила коммуникации

- Соблюдаем орфографию и оформление - маркдаун, единый регистр
- В IT принято общаться на **ты** между коллегами - излишнее «вы» и академичный язык воспринимаются тяжело
- Капслок и множество восклицательных знаков = крик. Так не стоит
- Относимся к участникам так, как хотим чтобы относились к нам
- Позитивное общение и эмодзи - мелочь, но приятно и ценится
- Не демотивировать, а быть хорошим примером
