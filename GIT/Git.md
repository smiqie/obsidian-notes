## 2. Настройка Git

### 2.1 Создание учетной записи

Необходимо для идентификации автора изменений, при отправке на удаленный репозиторий (чтобы другие разработчики знали от кого пришли изменения)

```bash
git config --global user.name "smiqie"
git config --global user.email "smiqie9201@gmail.com"
```

**Флаги:**

- `--local` - настройки только для текущего репозитория (применяются к одному проекту). Команда выполняется внутри папки репозитория

- `--global` - настройки для всех репозиториев текущего пользователя ОС (применяются ко всем вашим проектам)

- `--system` - настройки для всех пользователей системы (требует sudo). Если на компьютере несколько учетных записей ОС, эти настройки будут применяться для всех

**Приоритет настроек:** local > global > system (локальные переопределяют глобальные, глобальные переопределяют системные)

**Пример использования:**

```bash
# Глобальная настройка для всех проектов
git config --global user.name "smiqie"

# Локальная настройка для конкретного проекта (переопределит глобальную)
cd /path/to/project
git config --local user.name "work_account"
```


**Просмотр настроек:**

```bash
git config --list # Отображает наши учетные записи, которые будут отпарвляться на удаленный репозиторий
```

## 3. Работа с локальным репозиторием

### 3.1 Инициализация и базовые команды

```bash
git init                    # Инициализация Git в текущей папке (превращает обычную папку в локальный репозиторий)
git status                  # Текущее состояние репозитория (показывает какие файлы изменены + какие входят в коммит)
```

### 3.2 Создание коммита

```bash
git add .                   # Добавить все файлы в staging area
git add css/                # Добавить конкретную папку
git add index.html          # Добавить конкретный файл

git commit -m "comment"     # Создать коммит с комментарием (флаг -m "comment" - добавление коммента)
```

- `Staging area` - промежуточная область, куда добавляются файлы перед коммитом с помощью `git add`

- **Важно:** изучить правила написания commit messages (Conventional Commits)

- Можем создавать сколько угодно коммитов, прежде чем отпарвить их в удаленный репозиторий

### 3.3 Просмотр истории

```bash
git log                     # Список всех коммитов
git log -p                  # Подробная информация о коммитах (с изменениями)
git log -3                  # Последние 3 коммита (или другое число)
```

### 3.4 Удаление файлов

```bash
git rm file.txt             # Удалить локально файл и добавить изменение в staging
git rm -r folder/           # Удалить локально папку (рекурсивно!)
```

**Зачем `git rm` вместо обычного `rm`?** `git rm` удаляет файл и автоматически добавляет это изменение в staging area. При использовании обычного `rm` нужно дополнительно делать `git add`.

## 4. Работа с удаленным репозиторием

### 4.1 Создание репозитория на GitHub

1. New Repository
2. Заполнить:
    - **Repository Name** - имя репозитория
    - **Public/Private** - публичный (все видят + могут просматривать файлы) / приватный (только вы и соавторы)
    - **README.md** - описание проекта (лучше не добавлять при первом создании, см. раздел 6.1 **необязательно добавлять**)
    - **.gitignore** - файл с правилами игнорирования файлов (можно добавить позже **необязательно добавлять**)
    - **License** - лицензия проекта (можно добавить позже **необязательно добавлять**)

### 4.2 Подключение локального репозитория к удаленному

```bash
git init                                    # Инициализация
git add .
git commit -m "Initial commit"

git branch -M main                          # Переименовать ветку master в main
git remote add origin https://ссылка        # Подключение к удаленному репозиторию
git remote -v                               # Проверка подключения (показывает URL, что за fetch/push?)

git push -u origin main                     # Отправка в удаленный репозиторий
```

**Пояснения:**

- `origin` - имя удаленного репозитория (стандартное название), по сути равен ссылки идущей после `origin == https://ссылка `
- `main` - название ветки в которую отправляем репо
	- у любого репозитория всегда есть несколько веток, и `main` - это итоговый оттестрируемый проект
- Флаг `-u` (или `--set-upstream`) - устанавливает связь между локальной и удаленной веткой, после чего можно использовать просто `git push`. Писать необязательно?

### 4.3 Клонирование репозитория

```bash
git clone https://ссылка-на-репо           # Скачать репозиторий со всеми коммитами (папка с репо скачивается в тот каталог, в котором мы открыли терминал)
git clone --depth 1 https://ссылка         # Скачать только последний коммит
```

При клонировании автоматически:

- Создается папка `.git`
- Устанавливается связь с удаленным репозиторием (не нужен `git remote add`)

### 4.4 Работа с конкретными коммитами

```bash
git checkout <hash>                        # Переключиться на конкретный коммит по его хэш-коду
```

**Примечание:** `<hash>` - это идентификатор коммита из `git log`

## 6. Решение типичных проблем

### 6.1 Конфликт при первом push (если создали README на GitHub)

При создании удаленного репозитория с файлом README автоматически создается первый коммит (т.к. репозиторий уже не пустой). При попытке push возникнет конфликт, т.к. локальный и удаленный реоп имеют разные файлы

**Решение:**

```bash
git pull origin main --rebase              # Объединить удаленный и локальный репозитории
git push origin main                       # Отправить изменения
```

**Пояснение:** `--rebase` перестраивает историю коммитов, помещая ваши локальные коммиты поверх удаленных.

# 4 Редактирование одного репозитория с разных компов

- Заходим в репозиторий (на гите) --> Settings --> Colloborators --> Add People
## 4.1 Эмуляция работы пару человек

1. `1й разработчик`: внёс изменения в удаленный репо
2. `2й разработчик`: загрузил все изменения с удаленного репо на наш локальный репозиторий, созданные другим разработчиком. Изменил локальный репо --> отправил на удаленный
3. при работе нескольких разработчиков, удаленный репо всегда будет отличаться с локальным (т.е. будут конфликты при разных коммитах)
	- исправление: нужно уметь сливать несколько коммитов в один (`merge commit...` - нужно выполнять когда есть отличия)
