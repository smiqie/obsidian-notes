# Variables

---

# Объявление переменных

```js
var x = 10;    // ❌ устаревшее, НЕ ИСПОЛЬЗОВАТЬ
let y = 20;    // ✅ локальная, можно переприсвоить
const z = 30;  // ✅ локальная, нельзя переприсвоить (константа)
```

- `var` - устаревший стандарт, использует hoisting (всплытие), глобальная видимость 
- `let` - переменная, значение можно менять
- `const` - константа, значение менять нельзя

**Лучшая практика:** всегда использовать `const`, менять на `let` только при необходимости (защита кода)

# Память: примитивные и ссылочные типы

```js
// Примитивные типы - хранятся в стеке
let num = 10;
let str = 'text';

// Ссылочные типы - хранятся в куче
const person = { name: 'Alex' };
const arr = [1, 2, 3];
```

- переменная при инициализации выделяет слоты в ОЗУ под значение
- **примитивные типы** хранятся в стеке (значение напрямую)
- **ссылочные типы** хранятся в куче (переменная хранит ссылку на значение)
- автоматический сборщик мусора освобождает место под неиспользуемые переменные

---

# Инициализация

```js
// Объявление (декларирование)
let x;

// Присваивание
x = 4;

// Объявление + присваивание (инициализация)
let y = 20;
```

- `const` **должна быть инициализирована** сразу при объявлении

```js
const z = 30; // ✅
const w;      // ❌ ошибка
```

# Область видимости

```js
// Блочная область видимости (let, const)
{
    const x = 100;
    console.log(x); // 100
}
console.log(x); // ❌ ошибка - x не существует

// Глобальная область видимости (var)
{
    var y = 200;
}
console.log(y); // 200 (работает, но плохая практика)
```

- `let` и `const` имеют **блочную** область видимости `{}`
- `var` имеет **функциональную** и **глобальную** область видимости

# Правила написания (best practices)

## Каждая переменная на отдельной строке

❌ **Плохо** (неудобно читать):

```js
const name='Max', age=28, isDeveloper=true;
```

✅ **Хорошо** (каждая переменная на отдельной строке):

```js
const name = 'Max';
const age = 28;
const isDeveloper = true;
```

# Правила именования (naming conventions)

## camelCase - для переменных и функций

```js
let isEnabled = true;
let userName = 'Alex';
```

## PascalCase - для классов и конструкторов

```js
class UserAccount {}
```

## SCREAMING_SNAKE_CASE - для констант

```js
const API_KEY = '123456';
const MAX_SIZE = 100;
```

# Что можно использовать в именах

```js
let name123;      // ✅ буквы и цифры
let user_name;    // ✅ нижнее подчеркивание
let $value;       // ✅ знак доллара
let firstName;    // ✅ camelCase

let 123name;      // ❌ не может начинаться с цифры
let user-name;    // ❌ дефис запрещен
```

**Правила:**

- буквы (a-z, A-Z)
- цифры (0-9) - но не в начале
- нижнее подчеркивание `_`
- знак доллара `$`

# Осмысленные имена переменных

✅ **Хорошо** (понятно что хранит переменная):

```js
const userName = 'Alex';
const userAge = 25;
const isActive = true;
```

❌ **Плохо** (непонятно что хранит переменная):

```js
const a = 'Alex';
const x = 25;
const flag = true;
```

**Рекомендации:**

- имена переменных должны быть **существительными** (userName, userAge)
- boolean переменные начинаются с `is`, `has`, `can` (isActive, hasAccess, canEdit)

# Динамическая типизация

```js
let value = 'text';  // строка
value = 40;          // теперь число
value = true;        // теперь boolean
```

- в JavaScript есть **динамическая типизация**
- не нужно указывать тип переменной
- тип определяется автоматически по значению

**Не стоит злоупотреблять!** Лучше создать новую переменную чем менять тип существующей.

# Присваивание `null` и `undefined`

```js
let someName = 'James';

someName = undefined; // ❌ так НЕ делают
someName = null;      // ✅ так делают (сброс значения)
```

- `null` - используется для **явного** сброса значения
- `undefined` - возникает **автоматически** когда значение не присвоено

---

# Hoisting (всплытие) - только для `var`

```js
console.log(x); // undefined (не ошибка!)
var x = 10;

// Код интерпретируется как:
var x;
console.log(x); // undefined
x = 10;
```

- `var` использует hoisting - переменная "всплывает" наверх при компиляции
- `let` и `const` hoisting не используют
- еще одна причина **не использовать `var`**
